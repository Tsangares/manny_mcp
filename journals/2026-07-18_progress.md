## The goal, restated

North star (unchanged, re-confirmed 2026-07-18): LLM-driven OSRS automation — a library of
unattended **routines** (YAML step sequences over atomic commands) that LLM drivers run, monitor, or
author when none exists. There is no final state; the routines themselves are the product. This
campaign exists to (1) make the codebase maintainable enough to build on, and (2) prove the stack
can run unattended, starting with Tutorial Island as the first hands-off routine. Design implication:
routine-authoring ease is the metric that matters, not any single grind loop — see
`journals/TUTORIAL_ROUTINE_READINESS_2026-07-18.md` and
`journals/ROUTINE_CORPUS_STUDY_2026-07-18.md` for the full analysis of what stands between "an agent
drove it once" and "a YAML replays it unattended."

## Crash log

The session absorbed roughly four distinct hard-stop events, all attributed to power/hardware
instability rather than software (per the handoff's explicit "CRASH REALITY: cause is POWER —
NOT software/OOM"):

1. **~20:07** — full machine crash, ~5 minutes after dispatching three parallel refactor agents.
   `/tmp` wiped (fresh agent transcripts lost, but zero durable work — both repos were clean and
   pushed). Recovered via the standard Xvfb + client relaunch recipe; session tokens survived.
2. **~22:0x** — Claude session limit hit (a different class of hard-stop, not a machine crash) mid
   drive, right after root-causing DEFECT-1; that write-up was lost and had to be re-derived from
   `/tmp/runelite.log`.
3. **~01:20** — the Claude Code harness process itself restarted, taking the RuneLite client and
   Xvfb down with it (labeled "crash #3" in the handoff). Session tokens again survived.
4. **Between the chapel and magic tutorial sections** (~10:xx, ahead of Driver #5's run) — another
   machine crash; recovery smoke-tested 5/5, and the magic section's chicken-cast step turned out
   to have already been credited pre-crash.

**Durability discipline that made this a non-event:** handoff-doc-first (the live handoff at
`journals/REFACTOR_CAMPAIGN_HANDOFF.md` is the single source of truth, kept current after every
sub-step), commit-per-unit for crash-cheap work (source/docs — resumes from git for free), and
journal-per-section for crash-expensive work (live client drives — resumes at the last passed
section instead of from scratch). Every crash cost minutes of relaunch, not hours of redone work.

**Parked idea:** migrate the client to `diort` (a LAN host at 10.0.0.x, residential IP) if it proves
more power-stable than the current box — avoids the crash cycle without introducing datacenter-IP
ban risk the way a cloud VM would. Explicitly deferred until after the refactor.
