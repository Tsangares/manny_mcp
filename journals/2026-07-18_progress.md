## Tutorial Island: cleared, then driven unattended

**The drive.** Five driver agents (#1-#5), spanning two jars (`5bd303e` then `2fcb602`) and three
mid-session crashes, took account `new` from character creation through all ten tutorial sections
to "Welcome to Lumbridge!" — **Tutorial Island acceptance test PASSED, end-to-end**
(`journals/TUTORIAL_TEST_DEFECTS_2026-07-17.md`, Driver #5). This surfaced most of the defect list
below.

**The engine upgrade.** A driver agent manually walking through ten YAML sections isn't the goal —
an unattended *routine* replaying them is. Commit `1c63c42` gave `run_routine.py` the primitives it
was missing: `repeat_until` (check-first, capped iterations, per-iteration timeout — replacing a
buggy single-space-press loop), a narrow `dialogue`/`no_dialogue` predicate on `dialogue.open`
(fixes `await_condition:"dialogue"` fail-fast), `mcp_tool:click_text` wired to `CLICK_DIALOGUE`
(unblocks section 10's dialogue-heavy ending), and chain/directory support in the runner plus a new
`routines/tutorial_island/00_master.yaml` stitching sections 01→10. 34 new tests, 133 total passing.

**The live test.** With the engine upgraded, sections were run for real, observe-only, no manual
driving: **sections 01, 02, and 03 completed fully unattended** (character creation, Gielinor
Guide, Survival Expert fishing all clean SUCCESS), and section 04 got through woodcutting (+25 XP)
and firemaking (+40 XP) before **genuinely sticking** on the cook-shrimp step — reproduced twice,
identical failure (`journals/TUTORIAL_ROUTINE_LIVE_TEST_2026-07-18.md`). Root cause: a
**routine-content timing gap, not an engine or plugin bug** — the cook command fired ~5s after
tinderbox-on-logs, before the Fire GameObject had actually spawned. The same run also caught a
real bug: the account was found already disconnected on arrival, and the engine's crash detector
mismisread the resulting stale state file as a client crash.

**Two fixes landed same-day:**
- `40f213d` — cook-shrimp step now uses `repeat_until: no_item:Raw shrimps` (7s/attempt, 6 max) so
  it retries until the fire exists; added a gate-refusal-modal dismiss step and a one-shot
  re-entry recovery step, both no-ops on the success path.
- `1a4b5da` — `run_routine.py` now threads a real `MultiRuneLiteManager` into `routine.py` (it was
  silently getting `None` before, from a 2-arg vs 3-arg `set_dependencies` mismatch, so
  `_auto_restart_client()` always short-circuited), and added `_get_connection_status()`, which
  queries the plugin's own `GET_GAME_STATE` command as an authoritative discriminator between
  "actually crashed" and "disconnected / on login screen" — replacing the old mtime-staleness-only
  heuristic (see `journals/ENGINE_DISCONNECT_RECOVERY_SPEC.md`).

Both fixes are spec-complete and committed but **not yet live-verified** — that's queued for the
routines phase, on `newbakshesh`, which is parked mid-Tutorial-Island (~section 04) specifically as
this test bed.

## The goal, restated

North star (unchanged, re-confirmed by the user this session): LLM-driven OSRS automation — a
library of unattended **routines** (YAML step sequences over atomic commands) that LLM drivers run,
monitor, or author when none exists yet. There is no final state; the routines themselves are the
product. The design implication that falls out of this: **routine-authoring ease is the metric
that matters**, not any single grind loop or milestone. An LLM should be able to look at "no
routine exists for X" and write one about as easily as it can currently monitor one that already
exists. This campaign exists to (1) make the codebase maintainable enough to build routines and
tooling on top of, and (2) prove the stack can run unattended, with Tutorial Island as the proof
case. Full analysis in `journals/TUTORIAL_ROUTINE_READINESS_2026-07-18.md` and
`journals/ROUTINE_CORPUS_STUDY_2026-07-18.md`.

## The thermal saga

The session absorbed roughly **five distinct hard-stop events**. Early on these were chalked up to
power/hardware instability with "no software fix, just fast recovery" as the stated strategy. Later
in the session, a concrete root cause was found and it changed how the client gets operated for the
rest of the campaign:

**Root cause: heat, not power.** The RuneLite client pins ~79% of a CPU core *continuously* —
GPU-less software rendering on Xvfb renders every frame even when the character is completely idle.
Sustained load pushes the package to 77°C+ and the machine hard-stops. Measured this session:
**killing the idle client dropped package temperature 77°C → 72°C** and load average 1.45 → 0.81
within seconds.

**Thermal policy adopted** (`journals/REFACTOR_CAMPAIGN_HANDOFF.md`, "THERMAL POLICY"):
1. Client **off** during all source-refactor phases (J2-4/5/6/7/8, doc work, Python) — it isn't
   needed to edit or compile Java, only for live gates and routine tests.
2. Client **on** only for the duration of a gate or test, then killed again immediately after.
3. When it must run, it gets reniced to low priority right after launch (`renice 15 <pid>`;
   `cpulimit` isn't installed). Long routine-test sessions are meant to monitor package temp and
   pause above 88°C.
4. Concurrent subagents are not stacked while the client is running — agents + client together is
   the hot combination; source-phase agents (client off) are fine.

`scripts/client.sh` (commit `b91762c`) operationalized this as a lifecycle helper
(status/stop/start/restart) so every live-gate step in the refactor could cleanly bring the client
up, test, and tear it back down without ad-hoc process juggling.

**Parked idea:** migrate the client to `diort`, a LAN host on a residential IP, if it proves more
power/thermal-stable than the current box — would sidestep the crash cycle entirely without the
datacenter-IP ban risk a cloud VM would carry. Explicitly deferred until after the refactor
finishes.

**Durability discipline that made every crash a non-event:** the live handoff doc
(`journals/REFACTOR_CAMPAIGN_HANDOFF.md`) as single source of truth, kept current after every
sub-step; commit-per-unit for crash-cheap work (source/docs — resumes from git for free); and
journal-per-section for crash-expensive work (live client drives — resume at the last passed
section, not from scratch). Every crash this session cost minutes of relaunch, not hours of redone
work.
